# 检测辅助工具开发原则

## 一、 产品价值观（The Compass）
**“始于业务直觉，精于极致效率，忠于专业溯源，安于柔性防灾，形散神聚于解耦。”**

## 二、 终极验收标准（The 6 Golden Tests）
在开发任何新功能时，必须能在脑海中通过这六大极限测试：
1. **新手测试：** 一个刚入职的应届生，不看说明书，能否靠“示例模板”和“悬停动图”在 5 分钟内走通流程？
2. **极限测试：** 导入 500 页的报告、200 张图表或上万条沉降数据时，UI 是否绝对流畅（不假死），进度条是否精准？
3. **变更测试：** 甲方要求将所有报告图表的“红色实线”改为“蓝色虚线”，能否通过修改配置文件在 1 分钟内全局生效？
4. **溯源测试：** 最终生成的 Word 报告，能否反向追溯到具体哪一天的哪个原始设备导出文件？
5. **断电测试：** 软件运行中途直接强杀进程，重新打开时，临时快照能否恢复到灾难前一秒的状态？
6. **升级测试：** 国家出台了新的修约规范，能否只替换一个底层算法模块而不改动任何 UI 框架？

## 三、 七大核心开发原则（Product & Engineering Principles）

### 1. 业务直觉（贴合土木行规，说“人话”）
* **工程术语体系：** UI 界面和底层参数命名全面摒弃程序黑话。严禁出现“X轴数组”，必须使用“桩号/里程”；严禁出现“布尔值”，必须使用“是否贯通”。
* **开箱即用的行业标准：** 内置国标/地标默认参数。为常见的检测报告（如混凝土回弹、钢结构焊缝）提供“一键符合XX规范”的默认预设，做到**约定优于配置（Sensible Defaults）**。少数高阶用户才去动配置文件。

### 2. 极致效率（消灭重复劳动）
* **全场景批量化：** 任何涉及图表、计算、排版的功能，底层设计必须支持 List 级别的批量输入，拒绝单点循环操作。
* **高维联动与一键宏观：** 彻底打通“数据源（Excel/底层解析） -> 处理 -> 结果（Word/CAD）”的任督二脉。支持源数据更新后的图表自动重绘，并提供“一键清理空行”、“一键更新目录”等复合操作。

### 3. 降阻交互（极简与透明）
* **所见即所得（WYSIWYG）：** 复杂排版与出图前，强制进入轻量预览流。
* **新手保姆级引导：** 提供“生成示例模板（Demo Data）”按钮，一键导出带批注的 Excel 模板。复杂表单旁提供 `[?]` 悬停提示，极简科普操作逻辑。
* **模块化收纳：** 遵循 Win11 原生规范，按“数据接入 -> 核心处理 -> 报告导出”的真实工作流布局。高频功能平铺，低频功能折叠。

### 4. 柔性防灾（给用户发“后悔药”）
* **异步与状态可视：** 耗时 IO 和密集计算强制扔进后台线程（`QThread` 或 `asyncio`），前台 UI 必须保留清晰的进度反馈，彻底消灭“程序未响应”。
* **非破坏性与快照兜底：** 核心操作全面接入全局撤销（Undo）栈。高危覆盖操作前，后台自动在临时目录生成 `.bak` 备份文件。
* **“说人话”的异常捕获：** 拦截所有 Python 原生报错，转化为三要素提示弹窗：“**错在哪**（如：第5行C列标高数据） -> **为什么**（包含字母） -> **怎么改**（请修改为纯数字）”。

### 5. 专业严谨（敬畏工程规范）
* **算法白盒化：** 计算结果绝对精准，修约规则严格封装，杜绝浮点数精度丢失导致的不合规。必要时支持导出计算过程。
* **数据防篡改：** 原始数据（如万能机导出文件）与处理结果分离，建立完善的操作日志流水，经得起未来的工程倒查。

### 6. 柔性架构（拥抱变化）
* **绝对配置化：** 规范系数、图纸图框样式、Word 标签名等，全盘抽离至 `config.yaml` 或 `settings.json`，严禁在 `*.py` 中写死。
* **业务插件化：** 核心框架仅作为底座，具体的市政/房建检测项目按独立模块挂载，实现热插拔。

### 7. 克制解耦（高信噪比代码与 UI）
* **等宽数字与克制色彩：** 涉及工程数据的表格强制等宽字体，仅对超标/异常数据高亮预警，降低视觉疲劳。
* **严格的 MVC 分层：** UI 界面只负责画图和发信号，**严禁**在按钮回调事件中直接写读写 Excel 的逻辑。
* **契约编程：** 开启全量类型注解（Type Hints），复杂数据流转强制使用 `@dataclass`。
