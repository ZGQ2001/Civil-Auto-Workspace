# **检测辅助工具开发与重构原则总纲**

## **一、 业务背景与系统边界**

在土木工程领域（如道路桥梁、房屋检测与鉴定等），检测报告的编制一直是一个繁琐且容易出错的过程。

**系统边界明确：** 本工具的定位是\*\*内业报告自动化编排系统\*\*，并非底层原始信号处理软件。我们不替代仪器自带的专业分析软件，也不负责生产复杂的 CAD 图纸。

本工具的核心业务是：**接收仪器软件导出的现成 Excel/CSV 数据，聚焦于对 Word 和 Excel 进行高效的流转、规范套用与自动化排版。**

工具的底层逻辑和交互设计必须严格贴合以下标准内业工作流程：

**获取仪器导出数据 \-\> 导入本工具 \-\> 数据格式化与规范评定 \-\> 根据情况是否生成辅助图表 \-\> 基于模板自动填充报告 \-\> 人工校核 \-\> 自动化排版**。

本文档总结了我们在重构与开发过程中必须绝对遵循的核心原则、工程规范和验收标准。旨在确保每一个功能都能真正解决工程师的痛点，确保系统的高效、精准、以及底层架构的绝对健壮。

## **二、 终极验收标准（The 6 Golden Tests）**

在开发任何新功能时，必须能在脑海中通过这六大极限测试：

1. **新手测试：** 一个刚入职的应届生，不看说明书，能否靠“示例模板”和“悬停动图”在 5 分钟内顺畅走通上述标准内业流程？  
2. **极限测试：** 导入 500 页的报告、200 张图表或上万条沉降数据时，UI 是否绝对流畅（不假死），进度条是否精准？  
3. **变更测试：** 甲方要求将所有报告图表的“红色实线”改为“蓝色虚线”，能否通过修改配置文件在 1 分钟内全局生效？  
4. **溯源测试：** 最终生成的 Word 报告，能否反向追溯到具体哪一天的哪个原始设备导出文件？  
5. **断电测试：** 软件运行中途直接强杀进程，重新打开时，临时快照能否恢复到灾难前一秒的状态？  
6. **升级测试：** 国家出台了新的修约规范，能否只替换一个底层评定模块而不改动任何 UI 框架？

## **三、 十一大核心开发原则（Product & Engineering Principles）**

### **1\. 业务直觉（贴合土木行规，说“人话”）**

* **工程术语体系：** UI 界面和底层参数命名全面摒弃程序黑话。严禁出现“X轴数组”，必须使用“桩号/里程”等土木术语；严禁出现“布尔值”等 IT 词汇，必须使用“是否脱空”、“是否剔除异常值”等直观的工程判定词。  
* **开箱即用的行业标准：** 内置国标/地标默认参数。为常见的检测报告提供“一键符合XX规范”的默认预设，做到**约定优于配置（Sensible Defaults）**。

### **2\. 极致效率（消灭重复劳动）**

* **全场景批量化：** 任何涉及图表、计算、排版的功能，底层设计必须支持 List 级别的批量输入，拒绝单点循环操作。  
* **高维联动与一键宏观：** 彻底打通“Excel 数据源 \-\> 评定处理 \-\> Word 报告生成”的任督二脉。支持源数据更新后的图表自动重绘，并提供“一键清理空行”、“一键更新目录”等复合操作，辅助完成“排版”与“校核”环节。

### **3\. 降阻交互（极简与透明）**

* **所见即所得（WYSIWYG）：** 复杂排版与报告出图前，强制进入轻量预览流，便于人工校核。  
* **新手保姆级引导：** 提供“生成示例模板”按钮。复杂表单旁提供 \[?\] 悬停提示。  
* **模块化收纳：** 遵循 Win11 原生规范，完全按照标准内业流（数据接入 \-\> 格式化/评定 \-\> 报告生成 \-\> 校核 \-\> 终版输出）进行真实工作流布局。

### **4\. 柔性防灾（给用户发“后悔药”）**

* **异步与状态可视：** 耗时 IO 和密集计算强制扔进后台线程，彻底消灭“程序未响应”。  
* **非破坏性与快照兜底：** 核心操作全面接入全局撤销（Undo）栈。高危覆盖操作前，后台自动在临时目录生成 .bak 备份文件。  
* **“说人话”的异常捕获：** 拦截异常并转化为三要素提示：“**错在哪** \-\> **为什么** \-\> **怎么改**”。

### **5\. 专业严谨（敬畏工程规范）**

* **算法白盒化：** 评定结果绝对精准，修约规则严格封装，杜绝浮点精度丢失。  
* **数据防篡改：** 原始导入数据与处理结果分离，建立完善的操作日志流水备查。

### **6\. 柔性架构（拥抱变化）**

* **绝对配置化：** 规范系数、图表样式、Word 标签名等全盘抽离至 config.yaml 或 settings.json，严禁在 .py 中硬编码。  
* **业务插件化：** 核心框架仅作为底座，具体的道路、桥梁、房建等独立检测项目按独立模块挂载，实现热插拔。

### **7\. 克制解耦（高信噪比代码与 UI）**

* **严格的 MVC 分层：** UI 界面只负责画图和发信号，**严禁**在按钮回调中直接写读写 Excel/Word 的逻辑。  
* **等宽数字与色彩克制：** 涉及工程数据的表格强制等宽字体，仅对超标、异常数据高亮预警，降低校核报告时的视觉疲劳。

### **8\. 宿主环境防御（针对 Office 全家桶的防御编程）**

* **先检测后执行：** 在执行任何覆写保存操作前，必须先进行“文件占用锁”检测。如果文件被占用，UI 拦截并弹出友好的提示：“检测到 xxx.docx 正在被使用，请关闭后重试”。  
* **原子化保存：** 所有的处理和生成操作，先写入系统的临时目录（Temp），确认无误且目标路径未被占用后，再执行原子级的移动/重命名操作。

### **9\. 宽进严出的“脏数据”隔离区（Dirty Data Tolerance）**

* **设立 IO 隔离带：** 数据访问层（IO 层）读取仪器数据后，必须进行一次强力清洗（如去除前后空格、全角转半角、处理 NaN 和空行）。  
* **数据提纯：** 只有清洗干净、且成功装载进 models/schema.py 里的 @dataclass 结构体的数据，才被允许进入 core 层。核心算法（Core）绝对不相信外部数据。

### **10\. 隐式降级与醒目占位（防雪崩机制）**

* **局部失败不影响全局：** 遇到模板缺少标签、或者某个特定数据缺失时，不要中断整个生成流程。  
* **醒目占位：** 在日志（WARNING 级别）中记录该错误，并在 Word 对应位置强行插入一段红色高亮的文本（如 \[系统警告：此处缺少XXX数据\]），让校核人员一眼就能看到并手动补救。

### **11\. 数据处理的“幂等性”（Idempotency）**

* **输出一致性：** 无论点击执行多少次，针对同一份输入数据，产生的输出结果必须是完全一致的。绝对不能出现“每点一次，Word 报告就往后追加一段”的错误。  
* **状态清理：** 每次生成前，必须清理旧的上文状态或先清除目标区域的数据。

## **四、 强制性工程与编码规范**

在编写任何代码时，必须无条件遵守以下开发底线：

1. **架构解耦：** 核心业务逻辑（Core）、数据访问（IO）与用户界面（UI）必须完全分离。  
2. **契约与类型：** 全开类型注解（Type Hints）。模块间数据流转必须使用 @dataclass 结构体，**严禁裸传字典**。所有模块代码必须通过 Pylance Basic 级别的静态检查，严禁带红线提交。  
3. **安全与防错：**  
   * 资源必须用 with 管理。  
   * 严禁 except: pass 吞咽异常，报错必须带上下文变量。  
   * 系统命令安全：严禁拼接 Shell 字符串，必须用 subprocess.run(\[...\]) 列表形式。  
4. **跨平台与编码：**  
   * 必须使用 pathlib 处理所有路径。  
   * 所有文件读写必须显式指定 encoding='utf-8'。  
5. **日志与时区：**  
   * 时间处理必须带时区（如 UTC 转本地时区）。  
   * 严禁仅使用 print，关键节点必须用 logging 模块记录到文件。  
6. **依赖最小化：** 优先使用标准库解决问题，严禁为了极其简单的功能引入 pandas 等重型库。  
7. **分步交付：** 严禁一次性输出所有代码。所有系统设计必须分模块、分阶段推进。

## **五、 11项核心功能技术规范**

废弃原有 UI，全量替换为 **PySide6**，并使用 **qfluentwidgets** 库实现原生的 Windows 11 视觉交互。以下为具体功能的技术落地规范：

1. **工具分类树 \+ 搜索框：** 使用 qfluentwidgets 的 NavigationView 或 TreeWidget 实现。  
2. **工具详情页：** 采用多堆叠窗口（StackedWidget）或路由机制切换面板。  
3. **参数配置面板：** 基于 QFormLayout 动态读取 models/schema.py 中的配置结构自动生成输入框。  
4. **日志面板：** 基于 QPlainTextEdit，通过实现自定义的 logging.Handler，将不同级别（INFO/WARNING/ERROR）日志彩色高亮输出，支持 Ctrl+F 搜索和右键复制。  
5. **线程池：** **严禁在 UI 线程执行 IO 或密集计算**。必须基于 QThreadPool 和 QRunnable 实现，使用 Signal/Slot 与主线程通信。  
6. **预览窗口：** 支持动态挂载模块。图表预览使用 PySide6.QtCharts 或 matplotlib.backends.backend\_qtagg；表格预览用 QTableView。  
7. **表格编辑器：** 必须基于 QTableView \+ 自定义 QAbstractTableModel 实现，确保数据与视图分离。  
8. **撤销引擎：** 使用 Qt 的 QUndoStack 和 QUndoCommand 实现业务级操作（如表格数据修改）的撤销与重做。  
9. **批量队列：** 实现任务调度器。UI 端展示带进度的 QListWidget 或表格，支持通过全局标志位（Flag）向工作线程发送暂停/继续/跳过信号。  
10. **智能提示系统：** 若 Core 或 IO 层抛出自定义异常，UI 层需捕获并通过 qfluentwidgets 的 InfoBar 弹出友好的中文错误提示及修复建议。  
11. **进度与状态：** 底部挂载 QStatusBar 或使用环形/条形进度组件，必须与线程池的进度 Signal 绑定。
